# Context Map

Context Map là **bản đồ mô tả quan hệ và cách giao tiếp giữa các Bounded Context**. Nó cho thấy team nào phụ thuộc vào team nào, ai là upstream (nhà cung cấp), ai là downstream (người dùng), và cách tích hợp được thực hiện.

> Context Map là tài liệu strategic quan trọng nhất — nó cho thấy toàn cảnh hệ thống và chính trị giữa các team.

---

## 1. Tại sao cần Context Map

Khi hệ thống có nhiều BC, các câu hỏi cần trả lời:
- BC nào phụ thuộc vào BC nào?
- Khi BC upstream thay đổi API, BC downstream bị ảnh hưởng thế nào?
- Team nào có quyền quyết định model chung?
- Cách giao tiếp: sync hay async? REST hay event?

Không có Context Map → mỗi người hiểu integration theo cách riêng → inconsistency, bug khó tìm.

---

## 2. Các pattern trong Context Map

### 2.1 Partnership
Hai team **cùng cam kết** hợp tác chặt chẽ, thống nhất lịch release và thay đổi.

```
Flight BC ←──── Partnership ────→ Reservation BC
```

**Khi dùng:** Hai BC liên quan chặt, cùng một team hoặc team rất tin tưởng nhau.
**Rủi ro:** Một team block team kia nếu không phối hợp tốt.

---

### 2.2 Shared Kernel
Hai BC **chia sẻ một phần model chung** (code, schema). Mọi thay đổi trong phần chung phải được cả hai đồng ý.

```
Order BC ──┐
           ├── Shared: Money, Currency, CustomerId
Payment BC ─┘
```

```java
// Shared Kernel — package dùng chung
package com.airline.shared.domain;

public final class Money { ... }
public final class Currency { ... }
public final class CustomerId { ... }
```

**Khi dùng:** Hai BC cần consistency cao về một số type/concept.
**Rủi ro:** Coupling — thay đổi Shared Kernel ảnh hưởng cả hai BC.

---

### 2.3 Customer-Supplier (Upstream-Downstream)
**Upstream (Supplier)** cung cấp service/data. **Downstream (Customer)** phụ thuộc vào upstream.

```
Flight BC (Upstream) ──────────────────▶ Reservation BC (Downstream)
    [Supplier]                                [Customer]
```

Upstream có quyền quyết định API. Downstream phải adapt theo.
**Khi dùng:** Quan hệ phổ biến nhất giữa các BC trong cùng công ty.

---

### 2.4 Conformist
Downstream **chấp nhận hoàn toàn** model của upstream, không thay đổi, không wrap.

```
GDS (Global Distribution System — external) ──────▶ Reservation BC
          [Upstream, không thể thay đổi]              [Conformist]
```

**Khi dùng:** Upstream là hệ thống bên ngoài không thể thay đổi (vendor, partner).
**Hệ quả:** Downstream bị phụ thuộc chặt vào model của upstream — khó refactor sau.

---

### 2.5 Anti-Corruption Layer (ACL)
Downstream tạo **lớp dịch (translation layer)** để chuyển đổi model upstream sang model domain nội bộ.

```
Legacy Booking System ──▶ [ACL: Translator] ──▶ Reservation BC
                              ↑
                    Tách biệt legacy model
                    khỏi domain model mới
```

```java
// ACL — nằm ở infrastructure layer của Reservation BC
public class LegacyBookingTranslator {

    public Reservation translate(LegacyBookingRecord legacy) {
        return Reservation.reconstitute(
            new ReservationId(legacy.getBookRef()),
            new FlightId(legacy.getFlightCode()),
            mapStatus(legacy.getStatusCode()),    // "01" → HELD, "02" → CONFIRMED
            new Money(legacy.getFare(), Currency.VND)
        );
    }

    private ReservationStatus mapStatus(String code) {
        return switch (code) {
            case "01" -> ReservationStatus.HELD;
            case "02" -> ReservationStatus.CONFIRMED;
            case "09" -> ReservationStatus.CANCELLED;
            default -> throw new UnknownStatusException(code);
        };
    }
}
```

**Khi dùng:** Upstream có model xấu, legacy, hoặc không phù hợp với domain của mình.
**Lợi ích:** Domain nội bộ sạch, độc lập với upstream.

---

### 2.6 Open Host Service (OHS)
BC cung cấp **API chuẩn hóa, có tài liệu tốt** để nhiều consumer dùng được mà không cần customize riêng.

```
Flight BC (OHS)
    │
    ├──▶ Reservation BC
    ├──▶ Check-in BC
    ├──▶ Analytics BC
    └──▶ Mobile App
```

```java
// OHS — REST API chuẩn hóa của Flight BC
GET  /api/flights/{id}
GET  /api/flights?origin=HAN&destination=SGN&date=2025-06-15
GET  /api/flights/{id}/seat-availability
```

**Khi dùng:** Một BC phục vụ nhiều consumer khác nhau.

---

### 2.7 Published Language
Dùng **ngôn ngữ trung gian chuẩn hóa** (JSON Schema, Protobuf, Avro) để các BC giao tiếp mà không phụ thuộc vào implementation cụ thể của nhau.

```java
// Domain Event với Avro Schema — Published Language
{
  "namespace": "com.airline.events",
  "type": "record",
  "name": "ReservationConfirmed",
  "fields": [
    {"name": "reservationId", "type": "string"},
    {"name": "flightId", "type": "string"},
    {"name": "passengerId", "type": "string"},
    {"name": "confirmedAt", "type": "long"}  // epoch millis
  ]
}
```

Notification BC, Loyalty BC, Analytics BC đều nhận event này mà không cần biết Reservation BC được viết bằng gì.

## 4. Context Map và Team Topology

Context Map không chỉ là kỹ thuật — nó phản ánh **quan hệ giữa các team** (Conway's Law).

| Pattern | Team relationship |
|---------|------------------|
| Partnership | Hai team ngang hàng, phối hợp chặt |
| Customer-Supplier | Team A phụ thuộc Team B |
| Conformist | Team A chấp nhận toàn bộ quyết định của Team B |
| ACL | Team A tự bảo vệ mình khỏi ảnh hưởng của Team B |
| OHS | Một team phục vụ nhiều team khác |

---

## 5. Cách vẽ Context Map

Dùng ký hiệu đơn giản:
```
[BC A] ──U──▶ ──D── [BC B]
        (U=Upstream, D=Downstream)

[BC A] ──ACL──▶ [BC B]
[BC A] ←── Shared Kernel ──▶ [BC B]
[BC A] ──OHS──▶ [BC B], [BC C], [BC D]
```

Không cần tool phức tạp — whiteboard hoặc draw.io là đủ. Quan trọng là **cả team đều hiểu và đồng ý**.
