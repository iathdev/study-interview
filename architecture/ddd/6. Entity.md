# Entity

Entity là object có **định danh duy nhất (identity)** tồn tại xuyên suốt vòng đời của nó. Dù các thuộc tính thay đổi, entity vẫn là cùng một đối tượng chừng nào ID không đổi.

> Hai entity là một nếu chúng có cùng ID — dù mọi thuộc tính khác khác nhau hoàn toàn.

---

## 1. Đặc điểm

| Đặc điểm | Mô tả |
|----------|-------|
| **Có identity** | Được xác định bằng ID, không phải bằng giá trị thuộc tính |
| **Mutable** | Trạng thái có thể thay đổi theo thời gian |
| **Equality theo ID** | `entity1 == entity2` khi và chỉ khi `id1 == id2` |
| **Có vòng đời** | Được tạo, chuyển trạng thái, có thể bị xóa hoặc archive |
| **Rich behavior** | Chứa business logic, không chỉ là data container |
| **Tự bảo vệ invariant** | Tự kiểm tra rule trước khi thay đổi state |
| **Publish Domain Events** | Phát sự kiện khi có state transition quan trọng |

---

## 2. Identity — Định danh

ID là thứ duy nhất xác định một entity. Nó phải **không bao giờ thay đổi** sau khi được tạo.

```java
// Tệ — dùng primitive, dễ truyền nhầm tham số
void book(Long flightId, Long passengerId) { ... }
book(passengerId, flightId); // compile được nhưng sai nghiệp vụ

// Tốt — strongly-typed ID, compiler bắt lỗi ngay
void book(FlightId flightId, PassengerId passengerId) { ... }
book(passengerId, flightId); // compile error
```

### Chiến lược chọn loại ID

| Loại ID | Ưu điểm | Nhược điểm | Dùng khi |
|---------|---------|-----------|---------|
| **UUID** | Globally unique, sinh client-side | Dài, index kém hơn | Distributed system, event-driven |
| **Auto-increment (Long)** | Ngắn, index tốt, dễ đọc | Phụ thuộc DB, lộ thông tin | Monolith, internal entity |
| **Composite key** | Tự nhiên với domain | Phức tạp khi join | Entity phụ thuộc parent rõ ràng |
| **Natural key** | Có ý nghĩa business | Có thể thay đổi theo thời gian | Mã code cố định (mã sân bay, mã chuyến bay) |

ID phải là `private final` — một khi gán, không bao giờ thay đổi:

```java
public class Reservation {
    private final ReservationId id; // final — không có setter cho id

    public static ReservationId generate() {
        return new ReservationId(UUID.randomUUID());
    }

    public static ReservationId of(String value) {
        return new ReservationId(UUID.fromString(value));
    }
}
```

**Equality chỉ dựa vào ID** — hai reservation cùng id là một, dù mọi thứ khác khác nhau:

```java
@Override
public boolean equals(Object o) {
    if (!(o instanceof Reservation other)) return false;
    return id.equals(other.id);
}

@Override
public int hashCode() { return id.hashCode(); }
```

## 3. State & Vòng đời

Entity không chỉ tồn tại — nó **chuyển trạng thái** theo nghiệp vụ. Mỗi transition đều có rule riêng.

### Reservation lifecycle

```
         hold()              confirm()
[NEW] ──────────▶ [HELD] ──────────────▶ [CONFIRMED]
                     │                        │
              expire()│                cancel()│
                     ▼                        ▼
                [EXPIRED]              [CANCELLED]
```

### Flight lifecycle

```
                   delay()
[SCHEDULED] ──────────────▶ [DELAYED]
      │                          │
      └──────── depart() ────────┘
                    │
                    ▼
              [DEPARTED]
                    │
               arrive()│
                    ▼
               [ARRIVED]

[SCHEDULED / DELAYED] ──cancel()──▶ [CANCELLED]
```

## 4. Behavior — Không setter public

Entity **không được có setter public**. Thay vào đó, mọi thay đổi state đều đi qua method có tên nghiệp vụ rõ ràng. Method này tự kiểm tra invariant trước khi thay đổi — không để service bên ngoài làm thay.

```java
// Tệ — setter public, logic bị rò ra service bên ngoài
reservation.setStatus("CONFIRMED"); // ai cũng set được, không ai guard
reservation.setHeldUntil(null);

// Tốt — method nghiệp vụ, tự guard bên trong
public void confirm(Payment payment) {
    // Invariant kiểm tra ngay tại đây, không để service làm thay
    if (status != ReservationStatus.HELD) {
        throw new DomainException("Chỉ confirm được khi đang HELD, hiện tại: " + status);
    }
    if (isExpired()) {
        throw new DomainException("Reservation đã hết hạn");
    }
    if (!payment.getAmount().equals(totalPrice)) {
        throw new DomainException("Số tiền không khớp");
    }

    this.status = ReservationStatus.CONFIRMED;
    this.heldUntil = null;
    registerEvent(new ReservationConfirmed(id, flightId, passengerId, totalPrice));
}

public void cancel(CancellationReason reason) {
    if (status == ReservationStatus.CANCELLED) throw new DomainException("Đã cancel rồi");
    if (status == ReservationStatus.EXPIRED) throw new DomainException("Đã expired");

    this.status = ReservationStatus.CANCELLED;
    this.cancellationReason = reason;
    registerEvent(new ReservationCancelled(id, reason));
}

public void expire() {
    if (status != ReservationStatus.HELD) return; // idempotent
    this.status = ReservationStatus.EXPIRED;
    registerEvent(new ReservationExpired(id, flightId, seatId));
}

public boolean isExpired() {
    return status == ReservationStatus.HELD
        && LocalDateTime.now().isAfter(heldUntil);
}
```

## 5. Domain Events

Khi entity thực hiện một state transition quan trọng, nó **đăng ký Domain Event** để các BC khác biết và phản ứng — thay vì gọi trực tiếp sang BC khác.

```java
public abstract class BaseEntity {
    private final List<DomainEvent> domainEvents = new ArrayList<>();

    protected void registerEvent(DomainEvent event) {
        domainEvents.add(event);
    }

    public List<DomainEvent> pullEvents() {
        List<DomainEvent> result = List.copyOf(domainEvents);
        domainEvents.clear();
        return result;
    }
}
```

Entity chỉ **đăng ký** event — việc **publish** ra message bus là của Application Service, sau khi save thành công:

```java
// Application Service
public void execute(ConfirmReservationCommand cmd) {
    Reservation reservation = repo.findById(cmd.reservationId());
    reservation.confirm(cmd.payment());   // entity đăng ký event nội bộ
    repo.save(reservation);               // lưu trước

    reservation.pullEvents()              // lấy ra và publish
               .forEach(eventPublisher::publish);
    // → Notification BC gửi email
    // → Loyalty BC cộng điểm
    // → Analytics BC ghi nhận
}
```

## 6. Reference — Chỉ giữ ID, không giữ object

Entity chỉ được **tham chiếu đến Aggregate khác bằng ID**, không giữ object reference. Giữ object reference sẽ tạo coupling chặt và làm phức tạp transaction boundary.

```java
// Tệ — giữ object reference
public class Reservation {
    private Passenger passenger;  // load cả Passenger vào memory khi load Reservation
    private Flight flight;        // tương tự
}
reservation.getPassenger().getProfile().getLoyaltyPoints(); // Demeter violation

// Tốt — chỉ giữ ID
public class Reservation {
    private PassengerId passengerId;
    private FlightId flightId;
}
// Khi cần Passenger, gọi qua Repository
Passenger passenger = passengerRepo.findById(reservation.getPassengerId());
```

## 7. Anemic vs Rich Entity

**Anemic Entity (Anti-pattern)** — entity chỉ là data bag, business logic bị rò ra ngoài service:

```java
// Anemic — chỉ có getter/setter
public class Reservation {
    private String status;
    public void setStatus(String s) { this.status = s; }
    public String getStatus() { return status; }
}

// Logic bị rò ra service — sai
public class ReservationService {
    public void confirm(Reservation r, Payment p) {
        if (!r.getStatus().equals("HELD")) throw new Exception("...");  // guard ở ngoài
        if (r.getHeldUntil().isBefore(now())) throw new Exception("..."); // guard ở ngoài
        r.setStatus("CONFIRMED"); // set thẳng từ ngoài
    }
}
```

Hậu quả: guard bị bỏ sót khi có code path mới gọi `setStatus` trực tiếp, rule bị duplicate ở nhiều service, unit test phải test service thay vì domain object.

## 8. Ví dụ đầy đủ

```java
public class Reservation extends BaseEntity {
    private final ReservationId id;       // strongly-typed, final
    private final FlightId flightId;      // reference bằng ID
    private PassengerId passengerId;      // reference bằng ID
    private SeatId seatId;               // Value Object
    private ReservationStatus status;
    private Money totalPrice;            // Value Object
    private LocalDateTime heldUntil;
    private CancellationReason cancellationReason;

    // Tạo mới qua static factory — constructor private
    public static Reservation hold(FlightId flightId, PassengerId passengerId,
                                   SeatId seatId, Money price) {
        Reservation r = new Reservation();
        r.id = ReservationId.generate();
        r.flightId = flightId;
        r.passengerId = passengerId;
        r.seatId = seatId;
        r.totalPrice = price;
        r.status = ReservationStatus.HELD;
        r.heldUntil = LocalDateTime.now().plusMinutes(15);
        r.registerEvent(new ReservationHeld(r.id, flightId, seatId));
        return r;
    }

    public void confirm(Payment payment) {
        if (status != ReservationStatus.HELD) throw new DomainException("Không phải HELD");
        if (isExpired()) throw new DomainException("Đã hết hạn");
        if (!payment.getAmount().equals(totalPrice)) throw new DomainException("Sai số tiền");
        this.status = ReservationStatus.CONFIRMED;
        this.heldUntil = null;
        registerEvent(new ReservationConfirmed(id, flightId, passengerId, totalPrice));
    }

    public void cancel(CancellationReason reason) {
        if (status == ReservationStatus.CANCELLED) throw new DomainException("Đã cancel");
        if (status == ReservationStatus.EXPIRED) throw new DomainException("Đã expired");
        this.status = ReservationStatus.CANCELLED;
        this.cancellationReason = reason;
        registerEvent(new ReservationCancelled(id, reason));
    }

    public void expire() {
        if (status != ReservationStatus.HELD) return;
        this.status = ReservationStatus.EXPIRED;
        registerEvent(new ReservationExpired(id, flightId, seatId));
    }

    public boolean isExpired() {
        return status == ReservationStatus.HELD && LocalDateTime.now().isAfter(heldUntil);
    }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Reservation other)) return false;
        return id.equals(other.id);
    }

    @Override
    public int hashCode() { return id.hashCode(); }
}
```

## 9. Entity vs Value Object

| | Entity | Value Object |
|--|--------|-------------|
| **Identity** | Có ID duy nhất | Không có ID |
| **Equality** | So sánh bằng ID | So sánh bằng tất cả thuộc tính |
| **Mutability** | Mutable | Immutable |
| **Vòng đời** | Có — tạo, thay đổi, xóa | Không — replace bằng object mới |
| **Ví dụ** | Reservation, Flight, Passenger | Money, AirportCode, DateRange |
